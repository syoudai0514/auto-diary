# 0. 進め方とモデル引き継ぎプロトコル（最初に必ず読む）

実装の途中で担当AIモデルが変わる可能性があります（例: Fable 5 で開始し、途中から Opus 4.8 が引き継ぐ）。**会話コンテキストは引き継がれない前提**で、すべての状態をリポジトリ内の文書とコミットに残しながら進めてください。

## 0.1 Phase 0（計画）を最初にやり切る

実装に入る前に、リポジトリ・`docs/design/README.md`・本書を読み、**`docs/factnote/PLAN.md` を書いてコミット**してください。PLAN.md は「この会話を一切読んでいない別のAIが、PLAN.md と本書だけで実装を完遂できる」詳細度にします。含めるもの:

- 画面一覧と遷移（テキスト表現で可）
- データモデル最終版（§21 を出発点に確定させたもの）
- APIルート一覧（既存再利用 / 新規の区別付き）
- プロンプトファイル一覧と各プロンプトの要件
- 実装順序（§35 の P0 / P1 / P2）と各ステップの完了条件
- 再利用する既存コードの対応表（§1 を具体化したもの）
- 取り返しのつかない問題（セキュリティ・データ消失）になり得る論点があれば、ここで明示してユーザーに確認する

## 0.2 HANDOFF.md を常に最新に保つ

`docs/factnote/HANDOFF.md` を作業の区切りごと（最低でも各フェーズ末）に更新してコミットしてください。形式:

- 完了したこと
- 作業中のもの（ファイルと状態）
- 次にやること（順番付き）
- 検証コマンドと最新の実行結果
- 既知の問題・保留事項

## 0.3 コミットは小さく頻繁に

各フェーズ完了時に `npm run typecheck && npm run test && npm run lint && npm run build` を通してからコミットし、**常にビルド可能な状態を保つ**こと。セッションが途中で切れても、HANDOFF.md と直近コミットから再開できるようにします。

## 0.4 引き継いだAIへ

モデル切替後に引き継いだ場合は、まず `docs/factnote/PLAN.md` → `docs/factnote/HANDOFF.md` → 直近のコミットログの順に読み、会話履歴に頼らず再開してください。

## 0.5 トークン節約の指針

- `docs/design/*.dc.html`（巨大なHTMLプロトタイプ）は**開かない**。`docs/design/README.md` + `docs/design/screenshots/` + 実装済みの `tailwind.config.ts` / `globals.css` が正

**[更新]** 原案の「最初に必ず実施すること」14項目は調査済みです。以下を信頼してよい（不安があれば個別に差分確認）:

2. **パッケージマネージャー**: npm。scripts: `dev` / `build` / `start` / `lint` / `typecheck` / `test`(Vitest) / `test:e2e`(`node e2e/run.mjs`)。
3. **既存の画面と機能**: 音声日記（録音・テキスト入力・音声ファイル → 文字起こし → 日記生成 → AI修正 → 保存: コピー/共有/Appleジャーナル(設定でON時のみ)）、ふたりの話し合い分析（`/analyze`: 話者分離文字起こし → 構造化分析）、設定、プロフィール、ログイン/サインアップ。
4. **Gemini API**: `@google/genai`。**BYOK方式** — 各ユーザーが自分のAPIキーを設定画面で登録し、AES-256-GCM で暗号化して Upstash Redis に保存。全AI処理はサーバーAPIルート経由（キーはクライアントに一切出ない）。`src/lib/gemini.ts`（クライアント生成/モデル名/MIME推定/出力暴走の抑制）、`src/lib/aiRoute.ts`（キー解決 `resolveGeminiApiKey` とエラー応答 `aiErrorResponse` の共通化）。
5. **音声アップロード**: multipart/form-data。**Vercel の約4.5MBボディ上限**があるため、2.5MB超の音声はクライアントで 16kHz モノラル WAV にデコード・分割してから複数リクエストで送る（`src/lib/audioChunk.ts` の `expandToChunks`）。
6. **文字起こし**: `/api/transcribe`（通常）と `/api/talk/transcribe`（話者A/B分離）。`temperature: 0` + `maxOutputTokens` 上限 + 繰り返し畳み込み（`collapseRepeatedLines`）。
7. **構造化出力**: Gemini `responseSchema` + zod 検証 + JSONフェンス/波括弧抽出フォールバック + 1回だけ再生成、のパターンが `src/lib/generateDiary.ts` / `src/lib/analyzeTalk.ts` に実装済み。
8. **認証**: 招待コード制サインアップ、scrypt ハッシュ、HMAC署名Cookieセッション（`requireAuth` in `src/lib/apiAuth.ts`）。アカウントは Upstash Redis（`src/lib/userStore.ts`）。**日記等の本文はサーバーに保存しない方針**（本アプリも同じ）。
9. **レート制限**: `rateLimitDistributed`（Upstash固定ウィンドウ、Redis障害時はインメモリへフェイルオープン）。
10. **ローカル保存**: IndexedDB（`src/lib/drafts.ts`。自作の薄いラッパー。テストは fake-indexeddb）。
11. **PWA**: `public/manifest.webmanifest` + `public/sw.js` あり。
12. **テスト**: Vitest 31ファイル288件 + 自作Playwright E2Eランナー（`e2e/run.mjs`: 本番ビルド → モックUpstash(`e2e/mock-upstash.mjs`) → Gemini APIは `page.route` でモック → 4フロー）。
13. **デプロイ**: Vercel 想定（AI系ルートは `runtime='nodejs'` / `maxDuration=300`、Fluid Compute 前提）。
14. **デザイン**: 原案が参照するZIPと同一の「ウォーム・ペーパー」デザインシステムが**このリポジトリに実装済み**（`tailwind.config.ts` / `globals.css` にトークン反映、ライト/ダーク対応）。アイコンは `src/components/icons.tsx` の自作Lucide風インラインSVG（依存追加不要）。
15. **参考文書**: `docs/ROADMAP.md`。特に「柱3: 日記の保管庫（IndexedDB保存・一覧・検索・エクスポート）」は本アプリの記録一覧と重なるため、設計を統合してよい。

# 2. auto-diary で実際に踏んだ落とし穴と必須対策

**[更新]** 以下は本リポジトリの実運用で実際に起きた問題と対策です。**新機能にもすべて適用してください。**

1. **Vercel 4.5MBボディ上限**: 超えるとアプリのコードに到達する前にリクエストが拒否され、原因が分かりにくい。音声は `expandToChunks` の再利用で解決済み。**スクリーンショット画像にも同じ上限が適用される** → 画像はクライアントで長辺 ~1600px に縮小し JPEG/WebP 圧縮してから送ること（コスト削減も兼ねる）。
2. **並列デコードで iPhone のメモリ枯渇**: 複数音声を `Promise.all` でデコードすると実機Safariが落ちる。**必ず直列ループで処理**（`src/app/analyze/page.tsx` の `transcribeItems` が実例）。
3. **文字起こしの暴走（実際に起きた重大バグ）**: 3分の録音が4万字超の文字起こしになった。Gemini は無音・不明瞭区間で同一フレーズを際限なく繰り返すことがある。対策は `maxOutputTokens` 設定 + `collapseRepeatedLines`（`src/lib/gemini.ts`）。**画像抽出など新しい Gemini 呼び出しにも必ず出力上限を設定すること。**
4. **クライアント/サーバーのタイムアウト不一致**: クライアント90秒・サーバー300秒だと、サーバーが成功したのにクライアントが失敗扱い→再試行で二重課金になる。AI系リクエストはクライアント280秒（`src/lib/api.ts` の `AI_REQUEST_TIMEOUT_MS`）に統一済み。踏襲すること。
5. **無料枠の429**: `withRetryOn429`（`src/lib/retry.ts`）+ ユーザー単位レート制限を再利用。長い音声=多チャンク=処理に数分かかるため、**チャンク単位の進捗表示は必須**。
6. **iOSファイルアプリ経由のMIME欠落**: `file.type` が空や `application/octet-stream` になる。`guessAudioMimeType` で拡張子フォールバック済み。**画像入力でも同じ問題が起きる前提で実装すること。**
7. **モデル廃止**: gemini-2.0-flash 系は2026年6月に廃止された実績あり。モデル名はハードコードせず環境変数（`GEMINI_MODEL` / `GEMINI_TRANSCRIBE_MODEL`、既定 `gemini-3.1-flash-lite`）。
8. **構造化出力の失敗**: モデルがフェンス付きJSONや前置きを返すことがある。§1-7 の既存フォールバックパターンを再利用（新規実装しない）。
9. **decodeAudioData のアップサンプル**: 何もしないとデバイス既定レート（48kHz等）に変換されてサイズが逆に膨らむ。16kHzモノラルへの明示リサンプル（`decodeAudioBuffer`）を再利用。
10. **「全員に手動設定を配る」機能は失敗する**: Appleジャーナル連携は11ステップのiOSショートカット手動作成が必要で、配布相手に説明できず失敗した。教訓: **コア機能は誰でもゼロ設定で動くこと**。複雑な外部連携は設定で既定OFF + 有効化時に手順を表示する方式にする。
11. **E2EモックUpstash**: Redisクライアントは `enableAutoPipelining: false` で生成しないとモックサーバーと通信できない（既存コードは対応済み。新規にRedisクライアントを作る場合は踏襲）。
12. **ログ方針**: 音声・本文・画像・APIキーをログに出さない（既存ルールの継続。エラーログはタグ+ステータスのみ）。
13. **チャンク分割の副作用**: 分割すると音声のタイムスタンプと話者ラベルは**チャンクごとに独立**になる。§12.2 の「音声 02:14」のような根拠タイムスタンプはチャンクオフセットの補正が必要で、Gemini のタイムスタンプ自体も不正確なことがある → タイムスタンプ根拠は「参考値」とし confidence を併記。話者ラベルもチャンク間で入れ替わりうる（talk機能の既知の制約。分析プロンプト側で「内容の一貫性から補正する」指示を入れて対処している — `buildTalkAnalysisSystemPrompt` 参照）。

# 4. アプリの目的

このアプリは、家庭内や人間関係で起きた出来事について、音声・文章・録音ファイル・スクリーンショットなどを入力すると、AIが以下を分離して整理するアプリです。

- 原本から確認できる事実
- ユーザー本人の記憶や主張
- AIによる推測や解釈
- 現時点では判断できないこと
- 自分側の改善点
- 相手側の問題点
- 両者を踏まえた中立的な見解
- 次回の具体的な対応
- 相手に送る短い文章
- 保存用の日記

このアプリは、単にユーザーを肯定したり、相手を悪者として認定するアプリではありません。

本当の価値は、以下です。

> 自分が間違っている時は明確に指摘する。
> 相手の言動に問題がある時も曖昧にしない。
> 感情的な状況でも、事実や自分の感覚を見失わないようにする。

# 5. 想定する利用者

主な利用者は、以下のような悩みを抱えている人です。

- 夫婦や家族との会話で傷つくことが多い
- 相手の強い言葉に押され、自分が悪いのか分からなくなる
- 感情的な出来事を後から思い出せなくなる
- 自分の記憶や受け止め方に自信が持てない
- 相手を一方的に悪者にしたくはない
- 自分の改善点も知りたい
- 同じトラブルが繰り返されているか確認したい
- 日記や記録を簡単に残したい
- 相手への返信文を冷静に作りたい
- 子どもの前での衝突を減らしたい
- 離婚・別居・関係継続などの判断材料を、感情ではなく事実として残したい

# 6. 絶対に守るプロダクト原則

## 6.1 中立性

AIは、以下の回答をしてはいけません。

- ユーザーの話だけで相手の人格を断定する
- 相手を精神疾患や人格障害などと診断する
- 「相手は異常です」と断定する
- ユーザーを無条件に肯定する
- ユーザー側のミスを曖昧にする
- 両者を無理に50対50で悪いことにする
- 根拠がない出来事を事実として記録する
- 録音や文章に存在しない発言を補完する
- 安易に離婚や別離を勧める
- 相手を論破することを推奨する
- 復讐や嫌がらせを支援する

必ず以下を区別してください。

1. 原本で確認できる事実
2. ユーザーが記憶・主張している内容
3. AIによる推測
4. 判断材料が不足している内容

**[更新]** この原則群は `src/lib/prompt.ts` の `buildTalkAnalysisSystemPrompt`（率直な判定・人格でなく行動の評価・安全配慮）に近い実装が既にあります。ゼロから書かず、これを土台に強化してください。

## 6.2 具体性

抽象的な助言だけで終わらせないでください。

必ず、次回に実行できる具体的な行動を提示してください。

例：

- 依頼を受けた瞬間にリマインダーを登録する
- 「いつも」「全部」ではなく、今回の一件に限定して話す
- 車内や子どもの前では話し合いを中断する
- 「嫌だったことは分かった。ただ、その言葉は受け入れられない」と返す
- 生活費の追加精算を毎月決まった日に確認する
- 家事の担当を曖昧にせず、一覧にする
- 言った・言わないになった場合は、録音やメモを確認する
- 間違えて責めた場合は「勘違いだった、ごめん」と訂正する

次回の具体的対応は、原則として最大3件にしてください。

## 6.3 記録の信頼性

AIが生成した内容と、元の記録を混ぜないでください。

必ず以下を別データとして保存してください。

- 元の録音 / 元の音声ファイル / 元の文章 / 元のスクリーンショット
- AIによる文字起こし
- ユーザーが修正した文字起こし
- AI分析 / AI生成日記
- ユーザーが編集した日記
- AIが作った返信案 / ユーザーが実際に送った文章

AI分析や日記を再生成しても、原本を上書きしないでください。

## 6.4 心理的安全性

以下が含まれる場合は、通常の分析とは別に、安全確認カードを表示してください。

- 暴力 / 物を投げる / 身体的な威嚇 / 監禁 / 強い脅迫
- 自傷 / 他害
- 子どもへの暴力 / 子どもの安全に関する重大な内容
- 命に関わる発言 / 緊急性の高い危険

ただし、明確な危険がない場合に、毎回過剰な警告を出さないでください。

安全確認カードには、以下を表示してください。

- 今すぐ危険があるか
- 安全な場所へ移動できるか
- 信頼できる人へ連絡できるか
- 必要に応じて警察・救急・相談窓口を利用すること

**[更新]** `TalkAnalysisSchema` の `safetyNote`（暴力等の兆候がある場合のみ文言を返し、なければ空文字）と同じ方式が実装済みなので流用してください。また、録音の取り扱いについて、talk機能の導入画面にある「相手の同意を得てから」の注意書きに相当する簡潔な注意をオンボーディングに含めてください（過剰な警告にはしない）。

# 7. デザイン方針

## 8.1 今のことを話す

## 8.3 録音ファイルを読み込む

対応形式: m4a / mp3 / wav / mp4 / webm / aac（**[更新]** `src/lib/gemini.ts` の `EXT_MIME_MAP` が既にすべてカバー。これに加え caf / aiff / amr / flac / ogg / opus / wma も対応済み）

- 送信前にクライアントで長辺 ~1600px へ縮小 + JPEG/WebP 圧縮（Vercel 4.5MB上限とコスト対策。§2-1）

- **場所**: 自宅 / 車 / 外出先 / 公園 / 店舗 / 電話 / LINE / その他

# 10. 情報を3層に分けて保存する

## 10.1 原本

加工していない情報です。元音声 / 元画像 / 元文章 / 入力日時 / ファイル情報 / 録音時間 / ユーザーが入力した原文。**原本はAIが変更してはいけません。**

## 10.2 ユーザー本人の認識

ユーザーが記憶・主張している内容。「確認済みの事実」として扱わず、「ユーザー本人の認識」として保存してください。

## 10.3 AIの整理・推測

AIが整理した内容。必ず **確信度・根拠・判断できない点** を付けてください。

# 11. AI処理パイプライン

入力後は、原則として以下の順番で処理してください。

1. 音声または画像の解析
2. 音声の文字起こし
3. 発言者の仮分離
4. 日時・場所・人物の抽出
5. 事実と主張の分離
6. 不明点の抽出
7. 感情や論点の整理
8. 自分側の改善点の抽出
9. 相手側の問題点の抽出
10. 中立的見解の生成
11. 次回の具体的対応の生成
12. 相手への返信案の生成
13. 日記の生成
14. パターン分析用データの保存

処理画面では、現在どの段階かを表示してください（例: 「音声を文字にしています」「事実と解釈を分けています」）。

**[更新]** 実装形態: `/analyze` ページと同様に、クライアント側の状態機械からサーバーAPIルートを順に呼ぶ方式にしてください。ステップ 3〜13 は Gemini 呼び出し回数を抑えるため、「文字起こし（1呼び出し/チャンク）→ 構造化分析（1呼び出しで 4〜12 をまとめて生成）→ 日記生成（モード選択時に1呼び出し）」の3段構成を推奨します。

**文字起こしが完了した時点で必ず IndexedDB へ保存してから分析に進むこと。** ユーザーが途中キャンセルしても、分析が失敗しても、文字起こしと原本は失われないようにします（auto-diary のエラー画面「文字起こしは保持されています」と同じ思想）。

# 12. AI分析結果画面

分析結果は、以下の順番で表示してください。

## 12.1 最初の短い見解

一番上に、2〜4文程度で、率直かつ中立的な見解を表示してください。

例：

> 約束を忘れた点は、あなたのミスです。
> ただし、忙しかった背景や、他の家事・育児まで否定される理由にはなりません。
> 今回のミスと、人格や家庭への貢献は分けて考える必要があります。

曖昧な両論併記だけで逃げないでください。ユーザー側に明確な改善点がある場合は、明確に指摘してください。相手側の暴言や人格否定が明確な場合も、曖昧にしないでください。

## 12.2 確認できる事実

原本で確認できる内容だけを表示します。各項目に根拠を紐づけてください。

例: 音声 02:14 / 音声 04:32〜04:41 / 画像2枚目 / ユーザー入力 / 修正済み文字起こし

根拠をタップすると、該当する音声位置や画像を確認できるようにしてください。

**[更新]** 音声タイムスタンプの注意: チャンク分割された音声ではタイムスタンプにチャンクオフセットの補正が必要で、Gemini の返すタイムスタンプ自体も不正確なことがあります（§2-13）。タイムスタンプは「参考位置」として表示し、confidence を併記してください。MVPでは「音声(前半/後半)」程度の粒度に落とすことも許容します。正確にできないものを正確に見せないでください。

## 12.3 ユーザー本人の認識

確認済みではないものの、ユーザーがそう認識している内容です。

## 12.4 不明・確認できない点

例: ゴミ袋を実際に置いた人物 / 発言前の会話 / 相手の本当の意図 / 正確な発言 / その場にいた子どもがどこまで聞いていたか

## 12.5 自分側の改善点

人格評価ではなく、具体的な行動として表示してください。

- 悪い例: あなたは気が利かない / あなたは忘れっぽい人間だ
- 良い例: 受け取りを了承した時点でリマインダーを設定する / 追加生活費の確認を相手から言われる前に月1回実施する

## 12.6 相手側の問題点

相手の人格ではなく、具体的な発言・行動を対象にしてください。

例: 「死んでほしい」という表現を使用した / 一件のミスを「10年間ずっと」と一般化した / 誤認の可能性を指摘された後も訂正や謝罪をしなかった / 子どもの前で強い言葉を使用した

## 12.7 バランスの取れた結論

両者を無理に同じ割合で悪いことにしないでください。論点ごとに責任を分けてください。

> 冗談が不快だったことと、「死んでほしい」という表現が適切だったかは別問題です。
> ユーザーには冗談を控える改善点がありますが、存在を否定する言葉を受け入れる必要はありません。

## 12.8 次回の具体的対応

最大3件に限定してください。

## 12.9 相手へ伝える短文

以下の3種類を生成してください。

- **やわらかい**: 相手の気持ちを受け止めながら、自分の気持ちを伝える
- **標準**: 自分の反省点と、相手へのお願いを分けて伝える
- **境界線を明確にする**: 暴言や人格否定など、受け入れられないことを短く伝える

文章は短くしてください。人格批判、診断名、心理学用語を入れないでください。

## 12.10 日記を作成

モード選択: 事実記録 / 感情整理 / 家族日記 / 短い日記 / 詳細な日記。タイトルも自動生成。生成後にユーザーが編集・保存できるようにしてください。

# 13. 責任分解機能

「誰が何％悪い」という単一の数値は表示しないでください。代わりに、論点ごとに整理してください。

| 論点 | 自分側 | 相手側 | 判断 |
|---|---|---|---|
| 荷物の受け取りを忘れた | 改善が必要 | — | 自分側のミス |
| 強い暴言 | — | 改善が必要 | 不適切 |
| 忙しかった背景 | 考慮事情 | 配慮不足の可能性 | 断定不可 |
| 再発防止 | 通知設定 | 自分で受取可能な時間を指定 | 共同改善 |

評価ラベル: 明確な改善点 / 一部改善余地あり / 問題なし / 判断材料不足 / 相手側の改善点 / 共同で仕組み化 / 考慮事情あり / 原本から確認済み / ユーザー本人の認識

# 14. 認知・表現パターンの検出

複数の記録を横断して、以下のようなパターンを検出してください。

- 一般化表現（「いつも」「全部」「何もしていない」「10年間ずっと」）
- 一件のミスから人格全体を評価する / 過去全体へ話を広げる
- 相手を犯人と決めつける / 誤認判明後に訂正しない
- 謝罪を求めるが、自分は謝らない
- ユーザーがすぐ全面謝罪する / 自分の感情を飲み込む
- 約束や予定を忘れる / 口頭依頼だけで管理する
- 衝突の状況パターン（疲れている時 / 車内 / 出発前 / 子どもの前 / 実家関連 / お金 / 家事分担）

ただし、AIが心理的な診断をしてはいけません。あくまで「発言・行動パターン」として表示してください。

- 良い例: 「いつも」「全部」などの一般化表現が、今月5件の記録で確認されました。
- 悪い例: 相手は0か100かでしか考えられない人格です。

**[更新]** 実装方式: パターン検出は毎回全記録をAIに送るのではなく、各記録の分析時に `detectedPatterns` として構造化保存しておき、**横断集計はローカルで行う**（AI呼び出しなし・無料枠を消費しない・オフラインでも動く）。週次・月次の「講評文」生成のみAIを使う（P1以降）。

# 15. 良い出来事・修復行動の記録

悪い出来事だけを集計しないでください。以下も必ず記録・集計してください。

- 感謝された / 謝罪があった / 家事や子どもの対応を代わってくれた
- プレゼント・手紙 / 家族で楽しく過ごした / 外出・行事
- 仲直りの行動 / 言葉ではなく行動で埋め合わせた可能性
- 穏やかに話せた / 問題を仕組み化できた

「言葉では謝罪がなかったが、運転を代わった」などの場合は、事実と解釈を分けてください。

- **事実**: 相手がお酒を買い、帰りの運転を担当した。
- **ユーザーの解釈**: 埋め合わせの行動だった可能性があると感じた。
- **AIの見解**: 修復行動の可能性はあるが、意図は断定できない。

# 21. データモデル

最低限、以下のデータ構造を型定義してください。必要に応じて、より良い構造へ改善して構いません。

**[更新]** 原案のモデルに以下の変更を加えてください:

- 全レコードに `schemaVersion: number` を追加（将来のマイグレーション用）
- `Attachment.localUrl` は使わない。Blob を IndexedDB に直接保存し、表示時に `URL.createObjectURL` を都度生成・解放する（ObjectURL はセッション揮発のため保存する意味がない）
- `aiModel` / `promptVersion` は必須で記録（原案どおり）

# 26. 保存・バックアップ

このアプリは、非常にプライベートで重要な情報を扱います。

MVPでは、最低限以下を実装してください。

- IndexedDB へのローカル保存、再読み込み後の復元
- 録音中の下書き保存、入力中文章の自動保存、処理途中データの復元
- JSON一括エクスポート / テキスト一括エクスポート / Markdown一括エクスポート

**[更新]** 実装方式の指定:

- **PDF**: PDF生成ライブラリを追加**しない**こと。jsPDF等は日本語フォントの埋め込みが数MB規模になりPWAを肥大化させる。代わりに印刷用CSS（`@media print`）+ `window.print()` で「PDFとして保存」を実現する（iOS Safari の共有→プリント→PDF保存も案内）。

- **iOS退避リスク**: `navigator.storage.persist()` 必須（§21）。バックアップが唯一の確実な保全手段であることをUIで正直に伝える。
- 保存層はインターフェースで抽象化し、将来 iCloud Drive / Google Drive / 暗号化クラウド同期 / 独自バックエンドを追加できる構造にする（MVPで実装はしない）。
