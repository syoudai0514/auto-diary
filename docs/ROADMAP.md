# プロダクトロードマップ — 「話すだけで、整う」

このドキュメントは、音声日記アプリを次の段階へ進めるための**実装可能な詳細度のロードマップ**です。
この文書だけを読んで（過去の開発会話の文脈なしで）実装に着手できることを目的としています。
実装者（人間でもAIエージェントでも）は、着手前に必ず `README.md` と本文書の「共通の前提」を読んでください。

## プロダクトビジョン

**「書きたいのに書けない」をなくす。**

- 日記を書きたいが、書くのが苦手・面倒・話をまとめるのが苦手な人が、**ざっくばらんに話すだけ**（または録音済み音声を入れるだけ）で、整った日記と頭の整理を手に入れられる。
- さらに、夫婦・家族の**話し合いやケンカの音声**を入れると、話者ごとに言い分を整理し、**どちらの主張がより妥当かを率直に判定**し、双方の改善点・相互理解を支援する。「気持ちの翻訳者」になる。
- この価値が、アプリを開いた瞬間・人に紹介した瞬間に**伝わる**こと。

## 共通の前提（全施策で守ること）

1. **日記・会話の本文/音声はサーバーに一切保存しない**。保存はユーザーの端末内のみ（IndexedDB/localStorage）。サーバーが持つのはアカウント情報（ユーザー名・パスワードのハッシュ・暗号化されたGemini APIキー）だけ。
2. **BYOK**: Gemini APIキーはユーザーごとに自分のものを使う。新しいAIルートは必ず `requireAuth()` → `resolveGeminiApiKey()`（`src/lib/aiRoute.ts`）→ `getGemini(apiKey)` の順で書く。
3. **新しいAIルートの定型**: `runtime='nodejs'` / `maxDuration=300` / `rateLimitDistributed()`（`src/lib/rateLimit.ts`）/ 入力長の上限チェック / 失敗時は `aiErrorResponse()`。既存の `src/app/api/generate/route.ts` が手本。
4. **構造化出力の定型**: zodスキーマ（`src/lib/diary.ts` の `DiarySchema` が手本）+ Gemini `responseSchema`（`Type` enum）+ リトライ付き呼び出し（`src/lib/generateDiary.ts` の `runStructuredDiaryCall` が手本）。クライアント側でも応答をzodで再検証する（`src/lib/api.ts` の `parseDiaryResponse` が手本）。
5. **大きな音声はクライアント側で自動チャンク分割**（`src/lib/audioChunk.ts` の `expandToChunks`。Vercelのボディ上限約4.5MB対策）。429は `withRetryOn429`（`src/lib/retry.ts`）で自動リトライ。
6. **テストは必須**: 単体（vitest、既存の `*.test.ts` のモックパターンを踏襲）+ E2E（`e2e/` に実ブラウザフロー。`npm run test:e2e`。外部サービス不要で動くこと）。
7. デザインは「ウォーム・ペーパー」（クリーム地・墨色・テラコッタ）。画面部品は `src/components/screens/common.tsx`（CenterScreen/Chip/AutoTextarea）を再利用。

## 柱1: ふたりの話し合い分析 — ✅ 実装済み

> ステータス: 実装済み（2026-07）。以下は実装された仕様の記録。改善アイデアは末尾に。

**目的**: ケンカ・話し合いの音声から、感情的にならない第三者として「何が起きたか」「それぞれの本音」「どちらの主張がより妥当か（率直な判定）」「双方がどう改善できるか」「仲直りの入り口」を提示する。

**UXフロー**（`/analyze`、`src/app/analyze/page.tsx` の小さな状態機械）:
`intro`（機能説明＋**相手の同意を得て録音する**旨の明示）→ `input`（録音 or 音声ファイル）→ `transcribing`（話者付き文字起こし、チャンク進捗表示）→ `speakers`（文字起こしプレビュー＋「Aはだれ？Bはだれ？」自由入力、既定「私」「相手」）→ `analyzing` → `result` → `error`。

**結果画面のセクション**: ①率直な判定（`verdict.overall`＋A/B/五分五分、**人格でなく行動単位**の評価リスト）②それぞれの言い分・気持ち・本当の望み ③すれ違いポイント ④双方への改善提案 ⑤共通の願い ⑥仲直りの会話例 ⑦元の会話（折りたたみ）。全文コピー/共有で相手に見せられるサマリーを出力。**結果はどこにも保存しない**（揮発。保管は柱3と合わせて将来検討）。

**データスキーマ**（`src/lib/talk.ts` の `TalkAnalysisSchema`）:
`{ title, summary, topics[], sideA{label, claims[], feelings[], needs[]}, sideB{...}, misunderstandings[{point, aView, bView, explanation}], verdict{overall, leansToward:'A'|'B'|'even', behaviorsA[{behavior, assessment}], behaviorsB[...]}, adviceA[], adviceB[], commonGround[], reconciliationScript[{speaker, line}], safetyNote }`

**プロンプト安全設計**（`src/lib/prompt.ts`）: 文字起こしにない事実を補わない/話者ラベルの揺れは内容から補正/判定は率直に（偽りの五分五分にしない）が評価は常に発言・行動単位/暴力・脅迫の兆候があれば判定を控え `safetyNote` で専門窓口への相談を促す。

**API**: `POST /api/talk/transcribe`（音声→`A:`/`B:` 形式の話者付きテキスト）、`POST /api/talk/analyze`（話者付きテキスト＋話者ラベル＋プロフィール→構造化分析）。

**将来の改善アイデア**: 分析結果のPDF/画像書き出し（相手に渡しやすく）/ 時系列で「同じテーマのケンカが減ったか」の振り返り（柱3の保管庫に依存）/ 3人以上の会話対応。

## 柱2: 書けない人向け体験の強化（未実装）

**目的**: 「白紙を前に固まる」「話すことすら思いつかない」人を最後のひとりまで拾う。日記アプリとしての完成度を上げる。

### 2-a. 「きもちの整理」パネル（工数: 小、効果: 大。最初にやる）
生成される日記JSONには `facts`（事実）/`feelings`（気持ち）/`interpretations`（解釈）/`nextActions`（次にやること）が**既に含まれているのに、結果画面に表示されていない**。
- `src/components/screens/ResultScreen.tsx` に折りたたみセクション「きもちの整理」を追加し、4項目をアイコン付きリストで表示（空配列の項目は出さない）。
- 「頭の整理が苦手」ペルソナへの直接の回答。保存文面（`fullText`）に含めるかはユーザー設定（既定: 含めない）。
- テスト: ResultScreenは現状コンポーネント単体テストが無いので、E2Eのreviseフローに「きもちの整理が表示される」検証を1つ追加すれば十分。

### 2-b. インタビューモード（工数: 中）
- ホームの入り口に「質問に答えるだけ」を追加。AIが優しい質問を最大3問（例:「今日いちばん心が動いたのはどんな瞬間？」）→ 各質問に音声 or テキストで回答 → 回答をまとめて既存の `/api/generate` に渡す。
- 実装方式: 新ルートは不要にできる。質問は**固定プールからランダム**で開始し（APIコストゼロ・即表示）、回答後の深掘り1問だけ `/api/generate` 系の軽い呼び出しにする…のが複雑なら、v1は**固定質問3問のみ**で十分価値がある。回答は `質問: 回答` 形式で連結して transcript として渡す。
- 状態は `page.tsx` の状態機械に `interview` 画面を1つ足す（`quick` 画面の親戚。音声入力は既存の `useRecorder`+`/api/transcribe` を質問ごとに使う）。

### 2-c. 初回オンボーディング（工数: 小〜中）
- サインアップ直後のホームに「はじめの3ステップ」カード: ①Gemini APIキーを設定（未設定なら設定画面へ誘導。`getGeminiKeyStatus()` が既にある）②最初の日記を作る ③保存先を選ぶ。完了状態は localStorage。
- 現状の最大の離脱ポイントは「APIキー未設定のまま録音→エラー」。録音ボタン押下時に `hasKey=false` なら先に案内を出す方が親切（`/api/account/gemini-key` GETは軽い）。

### 2-d. ログイン/サインアップのミニLP（工数: 小）
- 未ログインで開いた人に価値が伝わるよう、ログイン画面上部に3枚カード:「話すだけで日記に」「頭とこころの整理」「ふたりの話し合い分析」。文言はREADME冒頭の売り文句と揃える。

## 柱3: 日記の保管庫（端末内のみ）（未実装）

**目的**: 現状は保存したら消える「変換器」。過去の自分に会える「日記の住まい」にする。継続率の核。

- **entriesストア**: `src/lib/drafts.ts` と同じIndexedDBパターンで別ストア `entries`（自動削除なし）。「保存する」実行時に下書きを保管庫へ昇格（外部アプリ保存と両立）。スキーマは `Draft` と同形＋`savedAt`。
- **一覧/検索/カレンダー**: 新ページ `/journal`。月カレンダー（記録がある日にドット）＋日別リスト＋全文検索（IndexedDBを全件読みしてクライアント検索で十分。数千件までは問題ない）。
- **気持ちの推移**: `feelings` と `tags` を週/月で集計した簡単なビュー（グラフライブラリは入れず、CSSバーで質素に）。
- **エクスポート/インポート**: 全件JSONをファイルとしてダウンロード/読み込み（機種変更・バックアップ）。**サーバー同期はしない**（プライバシー方針）。将来サーバー保管をやるなら必ずE2E暗号化＋明示オプトインで、方針転換としてREADMEに明記すること。
- ホームの「最近の記録」は保管庫と統合（下書き＝未保存、保管庫＝保存済み、をタブか見出しで区別）。
