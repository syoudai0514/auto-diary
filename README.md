# 音声日記PWA (Voice Diary) — 話すだけで、整う。

**書くのが苦手でも、話をまとめるのが苦手でも、日記は続けられる。** iPhoneに向かってざっくばらんに話すだけ（または録音済みの音声を入れるだけ）で、AIが**自然な日本語の日記**と**頭とこころの整理**にまとめてくれるPWAです。作成した日記は Apple純正「ジャーナル」・Day One・クリップボード・iOS共有シートへ保存できます。

- **話すだけ** — 録音するか、キーボードの音声入力で話すだけ。まとまっていなくていい。
- **AIが日記化** — 事実・気持ち・解釈を分けて、読みやすい日記本文に整えます。
- **ふたりの話し合い分析** — 夫婦・家族のケンカや話し合いの音声から、話者を聞き分けて言い分を整理。**どちらの主張がより妥当かを率直に判定**し、すれ違いの構造・双方の改善点・仲直りの会話例まで提示します（[詳細](#ふたりの話し合い分析)）。
- **どこにでも保存** — Appleジャーナル / Day One / コピー / 共有シート。
- **PWA** — iPhoneのホーム画面に追加すればネイティブアプリのように使えます。
- **招待制のアカウント** — 招待コードを持つ人だけがユーザー名・パスワードで登録できます。各自が自分のGemini APIキー（無料）を持ち込むため、運営側がAPIコストを負担する必要がありません（[招待制で複数人に使ってもらう](#招待制で複数人に使ってもらう)参照）。
- **プライバシー重視** — Gemini APIはサーバー側からのみ呼び出し、APIキーはブラウザに出しません。日記・会話の音声・本文はサーバーに保存もログ出力もしません（保存するのはアカウント情報のみ）。
- **無料で使える** — AIには各自のGoogle Gemini API無料枠を利用します（詳細は下記）。

「ウォーム・ペーパー」デザイン（クリーム地に墨色の文字、テラコッタのアクセント）をライト/ダーク両対応で実装しています。

## 技術構成

| 項目 | 内容 |
|---|---|
| フレームワーク | **Next.js 14**（App Router） |
| 言語 | **TypeScript**（strict） |
| スタイル | Tailwind CSS（デザイントークンをCSS変数で管理） |
| PWA | manifest + Service Worker + アイコン一式 |
| AI | Google Gemini API（無料枠あり、**サーバーサイドのみ**から呼び出し、各ユーザー自身のAPIキーを使用=BYOK） |
| 文字起こし | `gemini-3.1-flash-lite`（音声をインラインデータとして送信、環境変数で変更可） |
| 日記生成 | `gemini-3.1-flash-lite` + Structured Output（responseSchema） |
| 認証 | ユーザー名 + パスワード（scryptでハッシュ化）+ 招待制サインアップ + 署名付き HttpOnly Cookie |
| アカウント保存 | Upstash Redis（Vercel Marketplace連携）。保存するのはユーザー名・パスワードのハッシュ・暗号化したGemini APIキーのみ |
| 下書き保存 | IndexedDB（保存操作後も約24時間は端末内に退避、その後自動削除） |
| テスト | Vitest |
| デプロイ | Vercel（GitHub push で自動デプロイ） |

## 画面と使い方

- **ログイン/アカウント作成**: 既にアカウントがあればユーザー名・パスワードでログイン。招待コードを持っていれば「アカウントを作成」からユーザー名・パスワード・招待コードで新規登録できます（登録するとそのままログイン状態になります）。
- **ホーム**: 「録音して日記化」ボタン（マイク）、「すぐ話す（キーボード入力）」、「音声ファイルをアップロード」、「ふたりの話し合いを分析」の4つの入り口。未保存の下書きがあれば上部にバナー、下に最近の記録。
- **録音中**: 録音時間・波形・一時停止/再開・停止・キャンセル。20分を超えると注意バナー。
- **音声ファイルから作る**: ボイスメモなどで録音済みの音声ファイルを複数選び、選んだ順に1つの日記としてまとめられます（`<input type="file" multiple>` でOSのファイル選択画面から複数選択）。ファイルは一覧表示され、個別に削除・追加ができます。
- **処理中**: 音声を準備中（大きなファイルの自動分割、下記参照）→ 文字起こし中（複数件の場合は「n/m件目」の進捗表示）→ 日記生成中。
- **結果**: タイトル/本文を編集でき、「元の文字起こし」を折りたたみ表示。チップから Appleジャーナル/Day One/各種コピー/共有/**修正を依頼**/書き直す/削除。最下部の「保存する」が唯一の主要ボタン。
- **設定**: 文体（5種）・**プロフィール**（下記参照）へのリンク・**Gemini APIキー**（自分専用のキーを登録・更新。未設定の場合はここから登録しないと文字起こし・日記生成が使えません）・**Appleジャーナル連携**（既定オフ。準備手順を確認してから有効にするスイッチ。オフの間は保存先の選択肢にも出てきません）・標準の保存先（Day One/クリップボード/他の日記アプリを開く/毎回選ぶ、Appleジャーナル連携を有効にすればそれも追加）・Day Oneのジャーナル名・外観（端末に合わせる/ライト/ダーク）・ログアウト。
- **プロフィール**: 家族構成や自分の立場などをMarkdownで書き溜める専用画面（下記参照）。

日記は次の7通りで保存できます:
1. Apple純正「ジャーナル」へ iOSショートカット経由で保存
2. Day One へ URLスキーム経由で保存
3. **他の日記アプリを開く**（コピー+Shortcuts経由で起動、下記参照）
4. タイトルのみコピー
5. 本文のみコピー
6. タイトルと本文をまとめてコピー
7. iOS共有シートから他アプリへ送信

> **保存操作をしても下書きはすぐには消えません**: ショートカットの起動やURLスキームでの保存は、実際に相手のアプリ側で保存が成功したかをこのアプリからは確認できません（ショートカット内でエラーが起きても気づけません）。また、1つの日記を複数のアプリへ保存し直したい場合もあります。そのため、いずれかの保存操作をしても下書きはすぐに削除されず、ホーム画面の「最近の記録」に残り続けます（作成から**約24時間**で自動的に消えます）。不要になった下書きは「最近の記録」から手動で削除できます。

### プロフィール（家族構成・自分について）

家族との会話など、複数人が写る/話す内容を録音すると、AIだけでは「どの発言が自分（一人称）で、周りの人物をどう呼べばいいか」を正しく判断できないことがあります。設定画面の「プロフィールを編集」から、専用の**プロフィール画面**で以下のように管理できます。

- **現在のプロフィール**: Markdown形式のテキストとして表示・直接編集できます（例: `## 家族構成` の見出しの下に `- 妻(ママ)` のように箇条書き）。
- **情報を追加**: テキスト入力、または**マイクで話して**追加できます（録音→自動で文字起こし）。入力した内容と現在のプロフィールをAIが読み取り、矛盾なく統合した新しいプロフィール全文に更新します（「AIでまとめて追加」ボタン）。

例えば

> 私は4人家族の父です。妻はママ、子どもは長男・長女と呼びます。

と最初に登録し、後日「長女が幼稚園に入りました」と音声で追加すると、AIが両方の情報を保ったまま1つのプロフィールに整理してくれます。

このプロフィールは日記生成のたびに参考情報として使われ、一人称の判定や登場人物の呼び方（「相手」ではなく「妻」「ママ」など）に反映されます。空欄のままでも動作します（その場合は「私」「相手」で判定）。

> **プライバシーについて**: プロフィールは他の設定と同様、**端末内（localStorage）にのみ保存**され、サーバーには保存されません。プロフィールの内容は「情報を追加」でAIに整理してもらうときと、日記を生成するときにだけ、そのつどGeminiへ一時的に送信されます。このアプリは**過去の日記の内容を自動的に記憶することはありません**（サーバー・端末のどちらにも日記本体を保存しないため）。プロフィールに残したい特性や気づきがあれば、都度「情報を追加」から手動で登録してください。

### 日記の修正を依頼する（テキスト/音声）

一度生成した日記を見て「ここを直したい」と思ったときは、結果画面の「修正を依頼」チップから、生成し直さずにその場で直せます。

- チップを押すと入力欄が開き、**テキストで入力**するか、マイクボタンを押して**声で話す**かを選べます（声の場合は自動で文字起こしされ、入力欄に反映されます）。
- 例:「もっとカジュアルな文体にして」「ゴルフバッグの話は削って」「タイトルをもっと具体的にして」など、直したい内容を自由に伝えられます。
- 「この内容で修正する」を押すと、元の文字起こし・現在の日記・今回の指示をまとめてAIに送り、指示に沿って書き直された日記に画面上で置き換わります（下書きにも自動で反映されます）。この操作は何度でも繰り返せます。
- AIは修正の指示にない事実を勝手に作り足さないよう、元の文字起こしと今回の指示の範囲内でのみ書き直します。指示と関係ない部分はそのまま保たれます。

### ふたりの話し合い分析

ホームの「**ふたりの話し合いを分析**」から、夫婦・家族・パートナーとのケンカや話し合いを、感情的にならない第三者の目で整理できます。

1. **録音 or 音声ファイル**: その場で録音するか、録音済みの音声を選びます（長い会話も自動でチャンク分割して処理）。
2. **話者の聞き分け**: AIが声を聞き分けて `A:` / `B:` のラベル付きで文字起こしします。冒頭を確認して「Aはだれ？Bはだれ？」を入力してください。
3. **分析結果**:
   - **率直な判定** — どちらの主張がより妥当か、五分五分かを、理由付きではっきり示します（評価するのは常に発言・行動で、人格は評価しません）。
   - **それぞれの言い分・気持ち・本当の望み** — 言葉の奥にある「安心したい」「認められたい」を言語化します。
   - **すれ違いポイント** — 同じ出来事がそれぞれにどう見えていたか。
   - **双方への改善のヒント** — 明日から実行できる具体的な行動で。
   - **仲直りの会話例** — そのまま口に出せる短いセリフ。
4. **共有**: 全文コピー/共有シートで、相手に見せられるサマリーを出力できます。

> ⚠️ **必ず相手の同意を得て録音・アップロードしてください**。会話の音声・内容はサーバーに保存されず、分析結果も画面を閉じると消えます（残したい場合はコピーしてください）。会話に暴力・脅迫などの兆候がある場合、AIは優劣の判定を行わず、安全の確保と専門窓口への相談を促します。この分析はAIによる整理であり、最終的な判断はおふたりのものです。

### 事実ノート（独立アプリ・開発中）

`/factnote` 配下に、同一リポジトリで開発している**別プロダクト**「事実ノート — 感情的な出来事を、事実に戻す記録アプリ」があります。出来事の記録（文章・録音・音声ファイル）から、AIが「確認できる事実 / 本人の認識 / 推測 / 不明点」を分離し、自分側と相手側の改善点・論点別の責任整理・3種類の返信案・5モードの日記を生成します。記録はすべて端末内（IndexedDB）に保存され、サーバーには保存されません。

- 詳細仕様: `docs/FACTNOTE_REQUEST.md`（依頼書）/ `docs/factnote/PLAN.md`（実装計画）/ `docs/factnote/HANDOFF.md`（進捗）
- **お試し（外部サービス不要）**: `npm ci` のあと **`npm run dev:mock`** で起動 → `http://localhost:3000/factnote` を開き、任意のユーザー名・パスワード + 招待コード **`demo-invite`** でサインアップ。アカウントDBはメモリ上のモック（停止すると消える）、AIは固定のモック応答（`AI_MOCK=1`）です
- **アクセス**: ログイン後に `http://localhost:3000/factnote` を開く（既存アプリのホームからの導線は意図的にありません）
- **モックモード**: 通常の `npm run dev` でも環境変数 `AI_MOCK=1` を付ければ、Gemini APIキーなしで全フロー（文字起こし・分析・日記生成）を固定のモック応答で確認できます（開発・E2E用）
- **別デプロイ**: 本番は既存アプリとは別の Vercel プロジェクト + 別の Upstash Redis（`KV_REST_API_URL` / `INVITE_CODE` / `AUTH_SECRET` / `ACCOUNT_ENCRYPTION_KEY` をすべて別値に）で運用する想定です。その際 `NEXT_PUBLIC_APP_VARIANT=factnote` を設定すると `/` が `/factnote` にリダイレクトされます
- **サンプルデータ**: `/factnote/settings` から架空の10件を投入・削除できます
- **バックアップ**: 端末保存のみのため、設定画面のJSONエクスポートを定期的に実施してください（iOSは空き容量逼迫時にブラウザ保存データを削除することがあります）

## ローカルで動かす

前提: **Node.js 18.18 以上**（推奨 20 / 22）。

```bash
# 1. 依存関係をインストール
npm install

# 2. 環境変数ファイルを用意
cp .env.example .env.local
#   → .env.local を開いて AUTH_SECRET / INVITE_CODE / ACCOUNT_ENCRYPTION_KEY /
#     KV_REST_API_URL / KV_REST_API_TOKEN を設定
#   （Gemini APIキーは環境変数ではなく、サインアップ後にアプリの設定画面から登録します）

# 3. 開発サーバー起動
npm run dev
#   → http://localhost:3000 を開く

# その他
npm run build      # 本番ビルド
npm run start      # 本番モードで起動
npm run test       # テスト実行
npm run typecheck  # 型チェック
npm run lint       # ESLint
```

> マイク録音は **https もしくは localhost** でのみ動作します（ブラウザの仕様）。iPhone実機で試すときは Vercel の https URL を使ってください。

## 環境変数

`.env.example` を参照。Vercel では「Settings → Environment Variables」に同じキーを登録します。

| キー | 必須 | 説明 | 既定値 |
|---|:---:|---|---|
| `AUTH_SECRET` | ✅ | Cookie署名用のシークレット（32文字以上推奨）。 | — |
| `INVITE_CODE` | ✅ | 新規登録に必要な招待コード。共有したい相手にだけ伝える。 | — |
| `ACCOUNT_ENCRYPTION_KEY` | ✅ | アカウントに保存するGemini APIキーを暗号化する鍵（base64エンコードされた32バイト）。 | — |
| `KV_REST_API_URL` | ✅ | アカウント情報を保存するUpstash RedisのURL（Vercel Marketplace連携で自動注入）。 | — |
| `KV_REST_API_TOKEN` | ✅ | 同上のアクセストークン。 | — |
| `GEMINI_MODEL` | | 日記生成モデル。 | `gemini-3.1-flash-lite` |
| `GEMINI_TRANSCRIBE_MODEL` | | 文字起こしに使うモデル。 | `gemini-3.1-flash-lite` |
| `SESSION_DAYS` | | ログインの有効日数。 | `30` |
| `MAX_AUDIO_BYTES` | | 1リクエストあたりの音声アップロード上限（バイト）。 | `4194304`（4MB） |

> `NEXT_PUBLIC_` を**絶対に付けない**でください。付けるとブラウザに露出します。Gemini APIキーは環境変数ではなく、ユーザーごとにアプリの設定画面から登録し、暗号化してデータベースに保存されます（下記「Gemini APIキーの取得と設定」参照）。

`AUTH_SECRET` の生成例:

```bash
openssl rand -hex 32
```

`ACCOUNT_ENCRYPTION_KEY` の生成例:

```bash
openssl rand -base64 32
```

## Gemini APIキーの取得と設定（無料）

このアプリは **Google Gemini API** を使っています。**ユーザーごとに自分のキーを登録する（BYOK）**方式なので、アカウントを作った後、それぞれのユーザーが以下の手順で自分の無料キーを取得してアプリに登録してください（クレジットカード登録不要）。

1. <https://aistudio.google.com/apikey> にアクセスし、Googleアカウントでログイン。
2. 「**Create API key**」（APIキーを作成）をクリック。
3. 新しいプロジェクトを作るか、既存のプロジェクトを選ぶ（個人利用ならどちらでもOK。迷ったら「Create API key in new project」）。
4. 発行された `AIza...` から始まるキーをコピー。
5. アプリにログインし、**設定画面の「Gemini APIキー」欄**に貼り付けて「キーを保存」を押す。（暗号化してデータベースに保存され、以後そのアカウントでの文字起こし・日記生成に使われます）

> ⚠️ **無料枠の注意点**: Google AI Studio経由の無料利用では、送信した内容（音声・テキスト）が Google のサービス改善に利用される場合があります（有料版・Vertex AI経由では原則利用されません）。日記は個人的な内容を含むため、この点を理解した上でご利用ください。気になる場合は有料枠（Billingを有効化）に切り替えることもできます。
>
> 無料枠にはリクエスト数の**レート制限**があります（1分あたり・1日あたりの上限）。個人が1日に数回日記をつける用途では十分ですが、短時間に何度も試すと制限にかかることがあります（その場合はアプリ側にも簡易レート制限があるため、少し待ってから再試行してください）。
>
> **モデル名について**: Googleは無料枠のモデル名・レート制限を随時変更します（例: `gemini-2.0-flash`系は2026-06-01付けで廃止）。429エラーが頻発する場合は、[Google AI Studioの使用状況ページ](https://ai.dev/rate-limit)で現在のモデルごとの制限（RPM/RPD）を確認し、より枠の広いモデル名を `GEMINI_MODEL` / `GEMINI_TRANSCRIBE_MODEL` に設定してください。

利用モデルは環境変数で差し替えられます。既定の `gemini-3.1-flash-lite` は無料枠が広く、日本語の文字起こし・日記生成の両方に十分な性能があります。

## 招待制で複数人に使ってもらう

このアプリは**招待コードを持つ人だけ**が新規登録できます。誰でも自由に登録できるオープンなサービスではありません。

1. Vercelの環境変数 `INVITE_CODE` に、好きな合言葉のような文字列を設定する（例: `mochi-diary-2026`）。**これは全員で共有する1つのコード**です。専用の招待リンクや個別コードは今のところありません。
2. 招待したい相手（家族・友人など）に、アプリのURLと `INVITE_CODE` の値を伝える（LINEやメールなど、口頭以外の方法がおすすめです）。
3. 相手は `/signup`（ログイン画面の「アカウントを作成」リンク）から、好きなユーザー名・パスワードと、教えてもらった招待コードで登録します。**あなた自身の最初のアカウントも、同じ招待コードを使って同じ手順で作成します**（特別な手順は不要です）。
4. 登録が終わったら、各自が設定画面から**自分自身の無料Gemini APIキー**を登録します（[Gemini APIキーの取得と設定](#gemini-apiキーの取得と設定無料)参照）。運営側のAPIキーやコストを共有することはありません。

日記の本文・音声は今まで通り**各自の端末にしか保存されません**（サーバーに保存されるのはユーザー名・パスワードのハッシュ・暗号化されたGemini APIキーのみ）。

Appleジャーナル連携（[後述](#appleジャーナル保存用ショートカットの作り方)）は、iPhone側での一度きりのショートカット準備が必要なため、**招待した人ごとに既定でオフ**になっています。準備をしていない人がいきなり分かりにくいiOSのエラーに遭遇することはありません。Day One・クリップボード・共有シート・「他の日記アプリを開く」は準備なしで最初から使えます。

> **今のところ実装していないこと**: メールアドレスの確認、パスワードを忘れた場合のセルフサービスでのリセット機能はありません（少人数の招待制であることを前提にしています）。パスワードを忘れた場合は、Upstashのダッシュボードからアカウント情報を直接確認・修正する必要があります。招待コードを使い回すため、コードが広まってしまった場合はVercelの環境変数で値を変更してください（既存ユーザーのログインには影響しません。新規登録にのみ影響します）。

## Vercelデプロイ手順

1. <https://vercel.com/> に GitHub アカウントでログイン。
2. 「**Add New… → Project**」で先ほどの GitHub リポジトリを **Import**。
3. Framework Preset は自動で **Next.js** が選ばれます。そのまま **Deploy** せず、先にデータベースと環境変数を設定します。
4. **アカウント保存用のデータベースを追加する**: プロジェクトの「**Storage**」タブ（またはVercel Marketplace）から **Upstash** の Redis を追加します（無料枠あり）。追加すると `KV_REST_API_URL` / `KV_REST_API_TOKEN` が自動でプロジェクトの環境変数に登録されます。
   > Vercelのストレージ関連の製品名・手順は変更されることがあります。「Storage」または「Marketplace」内で "Redis" や "Upstash" を探してください。見つからない場合は [Upstash](https://upstash.com/) で直接Redisデータベースを作成し、発行された REST URL / TOKEN を手動で `KV_REST_API_URL` / `KV_REST_API_TOKEN` として登録しても構いません。
5. 「**Environment Variables**」で残りを追加（[環境変数](#環境変数)参照）:
   - `AUTH_SECRET`
   - `INVITE_CODE`
   - `ACCOUNT_ENCRYPTION_KEY`
   - （任意）`GEMINI_MODEL` / `GEMINI_TRANSCRIBE_MODEL` / `SESSION_DAYS` / `MAX_AUDIO_BYTES`
6. 「**Deploy**」を実行。数十秒で `https://<プロジェクト名>.vercel.app` が発行されます。
7. 以後、`main` ブランチへ push するたびに**自動で再デプロイ**されます。
8. デプロイが終わったら、まず**あなた自身が** `INVITE_CODE` を使って `/signup` からアカウントを作成し、設定画面で自分のGemini APIキーを登録してください。

> 文字起こし/生成APIは処理に時間がかかるため `maxDuration = 300`（秒）を指定しています。Vercelの実際の実行時間上限はプラン・Fluid Compute設定により異なるため、環境によっては届かないことがあります。それでもタイムアウトする場合は、音声を短く分けて（このアプリは大きな音声を自動チャンク分割します。[大きな音声ファイルの自動分割](#大きな音声ファイルの自動分割)参照）お試しください。

## Appleジャーナル保存用ショートカットの作り方

アプリの「Appleジャーナルに保存」ボタンは、iOSの**ショートカット**「**音声日記を保存**」を呼び出します。このショートカットを一度だけ作っておきましょう。**順番どおりタップするだけ**でできます。

> **準備が終わるまではAppleジャーナルの選択肢は表示されません**: このショートカットが無い状態で「Appleジャーナルに保存」を試すと、iOSの分かりにくいエラー（「ファイルが存在しません」）になってしまいます。それを避けるため、アプリの**設定画面に「Appleジャーナル連携」というスイッチ**があり、既定ではオフになっています。下のショートカットを作り終えてから、設定画面でスイッチをオンにしてください（オンにするまでは、保存先の選択肢にもAppleジャーナルは出てきません）。

> 用意するもの: iPhone（iOS 17.2以降・「ジャーナル」アプリが使えるもの）と「ショートカット」アプリ。

### ステップ（1アクションずつ）

1. **「ショートカット」アプリを開く。**（青いアイコン）
2. 右上の **＋（プラス）** をタップして、新しいショートカットを作る。
3. 上の名前の部分をタップして、名前を **`音声日記を保存`** にする。（※アプリと同じ名前にするのが超重要）
4. 「**アクションを追加**」をタップ。
5. 検索欄に「**ショートカットの入力**」と入れて、**「ショートカットの入力を受け取る」**（Receive Input）を選ぶ。
   - 「受け取る内容」に **テキスト** が含まれるようにする。
6. もう一度「**アクションを追加**」をタップ。検索欄に「**辞書**」と入れて、**「入力から辞書を取得」**（Get Dictionary from Input）を選ぶ。
   - このアクションの入力が、5でうけとった「**ショートカットの入力**」になるようにする。
7. また「**アクションを追加**」。「**辞書の値**」と検索して、**「辞書の値を取得」**（Get Dictionary Value）を選ぶ。
   - 「キー」に **`title`** と入力する。（これでタイトルが取り出せる）
   - できた結果に、右にある `>` などから名前をつけられるなら **「タイトル」** と名づけておくと分かりやすい。
8. もう1つ「**辞書の値を取得**」を追加。今度は「キー」に **`body`** と入力する。（これで本文が取り出せる）
   - 名前を **「本文」** にしておく。
9. 「**アクションを追加**」→「**ジャーナル**」と検索して、**「ジャーナル項目を作成」**（Create Journal Entry）を選ぶ。
   - 本文の欄に、8でつくった **「本文」** を入れる。
   - タイトルの欄があれば、7でつくった **「タイトル」** を入れる。
10. さいごに「**アクションを追加**」→「**通知**」と検索して、**「通知を表示」** を選び、文章に **`日記を保存しました`** と書く。（保存できたか分かるように）
11. 右上の **完了** をタップして保存。

これで完成です。アプリの結果画面で「Appleジャーナルに保存」を押すと、このショートカットが起動して、ジャーナルに新しい記録が追加されます。

### 渡されるデータについて（作る人向けメモ）

アプリは次のURLでショートカットを起動します:

```
shortcuts://run-shortcut?name=音声日記を保存&input=text&text={URLエンコード済みJSON}
```

`text` の中身（JSON）は次の形です:

```json
{
  "title": "タイトル",
  "body": "本文",
  "tags": ["タグ"],
  "createdAt": "2026-07-11T09:30:00.000Z"
}
```

ショートカットの手順は、この `text`（＝ショートカットの入力）を辞書に変換し、`title` と `body` を取り出してジャーナルに保存する、というものです。

### うまくいかないとき（長すぎる場合の代替）

日記がとても長いと、URLが長すぎてショートカットが起動できないことがあります。その場合アプリは自動で**内容をクリップボードにコピー**し、「貼り付けてください」と案内します。ショートカット側で「クリップボードを取得」してから辞書に変換するようにしておくと、この方法でも保存できます。

## 大きな音声ファイルの自動分割

Vercelのサーバーレス関数には、1リクエストあたり約4.5MBという**プラットフォーム側のボディサイズ上限**があります。これはアプリ側の設定では変更できません。そのため、これより大きい音声をそのまま送ると、サーバーに届く前に拒否されてしまいます。

このアプリは、選択・録音した音声が一定サイズを超える場合、**ブラウザ内で自動的に分割してから複数回に分けてアップロード**します。

- Web Audio API（`AudioContext.decodeAudioData`）で音声をいったんデコードし、モノラル・16kHz（話し声の文字起こしには十分な品質）に変換した上で、チャンクへ再分割します（コンテナ形式のファイルをバイト単位で単純に切ると壊れてしまうため、この方式を取っています）。
- 1チャンクあたりの目標サイズは約2.5MB（16kHzモノラルで約80秒ぶん）です。サーバー側の受け入れ上限（既定4MB）より意図的に小さくしており、**1回あたりのアップロード・文字起こしを速く終わらせてタイムアウトしにくくする**ためです。
- 分割された各チャンクは、複数音声ファイルアップロード機能と同じ仕組み（順番に文字起こし→結合）で処理されます。処理中は「音声を準備中…」→「文字起こし中…（n/m）」と表示されます。
- **元のファイルサイズではなく、実際の音声の長さで分割数が決まります**。M4A/AAC等の圧縮音声は、圧縮のおかげで見た目のファイルサイズが小さくても長時間の録音を含むことがあり、文字起こし用に非圧縮(WAV)へ変換するとデータ量が増えるため、圧縮ファイルの見た目のサイズから予想するより多くのチャンクに分かれることがあります（正常な動作です）。
- 音声の長さそのものには上限を設けていません（何分・何十分の録音でも、理論上は分割してアップロードできます）。ただし処理時間・Gemini無料枠のレート制限には影響するため、あまりに長い録音は分割してアップロードするか、複数回に分けて録音することをおすすめします。
- 1チャンクの処理に時間がかかる場合に備え、文字起こしのクライアント側タイムアウトは180秒、サーバー側の`maxDuration`は300秒に設定しています。

## 他の日記アプリとの連携（URLスキームが無いアプリ向け）

Apple純正ジャーナルやDay One以外にも、iPhoneには数多くの日記アプリがありますが、その多くは**外部アプリからの自動投稿に対応していません**（独自のURLスキーム・共有シート対応・Shortcutsアクションのいずれも持たない）。

そのようなアプリでも、結果画面の「**他のアプリを開く**」ボタン（または設定で保存先を「他の日記アプリを開く」に設定）を使うと、次の流れで手間を最小限にできます。

1. アプリが自動でタイトルと本文を**クリップボードにコピー**
2. あらかじめ作っておいたショートカット「**日記アプリを開く**」を自動起動
3. ショートカットが対象の日記アプリを開く
4. あなたが日記アプリ側で新規作成し、**貼り付け**て保存

### 「日記アプリを開く」ショートカットの作り方

1. iPhoneの「**ショートカット**」アプリを開く
2. 右上の **＋** をタップして新しいショートカットを作る
3. 名前を **`日記アプリを開く`** にする（アプリ側の設定と同じ名前にすること）
4. 「**アクションを追加**」→「**アプリを開く**」を検索して選ぶ
5. 開くアプリとして、普段使っている日記アプリ（例: シンプル日記など）を選ぶ
6. 右上の **完了** で保存

これだけです。以後、結果画面の「他のアプリを開く」を押すと、本文がコピーされた状態でそのアプリが開くので、新規作成して貼り付けるだけで保存できます。

> 対象アプリが独自のURLスキームや共有シートに対応しているか不明な場合は、まず「共有」ボタンから共有シートを開き、アプリの候補に出てくるか確認してみてください。出てくれば、そちらの方がさらに手軽です（コピー操作が不要になります）。iOSの「ショートカット」アプリの「アプリ」タブでアプリ名を検索し、そのアプリ専用のアクション（「エントリーを作成」等の具体的な名前のもの）が見つかれば、より直接的な連携ができる場合もあります。

## テスト

Vitest による単体・APIテストと、Playwright によるE2Eテストを用意しています。

```bash
npm run test          # 単体・APIテストを一度だけ実行
npm run test:watch    # 監視モード
npm run test:e2e      # E2Eテスト（本番ビルド → 実ブラウザで主要フローを検証）
npm run test:e2e -- --skip-build   # 既にビルド済みの .next を使って高速に再実行
```

### E2Eテストについて

`e2e/` ディレクトリの実ブラウザテストは、外部サービスに一切依存せずに動きます。

- **アカウントDB**: Upstash互換のモックサーバー（`e2e/mock-upstash.mjs`）をテスト中だけ起動
- **Gemini API**: 各フロー内でリクエストをモック（UIの配線・送信ペイロードの形を検証する）
- 初回は `npx playwright install chromium` でブラウザを取得してください（CI等でパスを指定したい場合は環境変数 `E2E_CHROMIUM_PATH`）

検証しているフロー: アカウント一連（招待コード→サインアップ→APIキー登録→再ログイン→未認証リダイレクト）、日記生成〜修正依頼、Appleジャーナル連携の有効化ゲート。

## セキュリティ上の注意

このアプリは招待制での少人数利用を前提にしていますが、公開URLを知られても第三者に使われないよう配慮しています。

- **認証**: ユーザー名・パスワードによるログイン。パスワードは平文で保存せず、**scryptでハッシュ化**（ランダムsalt付き）して保存します。成功時に**署名付き HttpOnly Cookie**（HMAC-SHA256、ユーザーIDを含む）を発行。`middleware.ts` で全ページ/APIを保護し、各APIでも多層的に検証します。
- **新規登録の制限**: 招待コード（`INVITE_CODE`）を知っている人しかアカウントを作成できません。
- **Gemini APIキーの暗号化保存**: 各ユーザーが自分のGemini APIキーを持ち込みます（BYOK）。サーバーに保存する際は **AES-256-GCM** で暗号化し、復号は各リクエスト処理中にサーバー側で一時的に行うのみです。平文のキーがクライアントへ返されることはありません。
- **保存しない**: 音声・日記本文はサーバー/Vercelに**一切保存しません**（文字起こし後に破棄）。サーバーログにも出力しません（エラーログにも本文を含めません）。サーバーに保存するのはアカウント情報（ユーザー名・パスワードのハッシュ・暗号化したGemini APIキー）のみです。
- **端末内のみ**: 未保存の下書き・設定・プロフィールは **IndexedDB / localStorage** に保存します。これらはサーバーには送信・保存されません。
- **レート制限**: ログイン・サインアップ・文字起こし・生成にレート制限を実装し、連続呼び出し・総当たり・招待コード推測を防ぎます。ログイン後の操作はIPアドレスではなく**アカウント単位**で制限されます。制限のカウントは**Upstash Redisで全インスタンス共有**（固定ウィンドウ方式）され、サーバーレスの複数インスタンス環境でも一貫して効きます（Redis未設定・障害時はインメモリ実装に自動フォールバックし、可用性を優先します）。
- **入力サイズ制限**: 1リクエストあたりの音声サイズ（既定4MB。Vercelのサーバーレス関数のリクエストボディ上限＝約4.5MBに配慮）・文字起こし長（2万文字）・パスワード長などに上限を設定。それを超える音声はクライアント側で自動的にチャンク分割される（下記参照）。
- **CSRF/XSS**: Cookieは `SameSite=Lax` / `HttpOnly` / 本番は `Secure`。表示はReactの自動エスケープに委ね、`dangerouslySetInnerHTML` は不使用。
- **オープンリダイレクト対策**: ログイン/サインアップ後に戻る `next` パラメータは、同一オリジンの絶対パスであることを検証してから使用します（`//evil.example` のようなプロトコル相対URLで外部サイトへ誘導されるのを防止）。
- **Gemini無料枠のデータ利用について**: Google AI Studio経由の無料利用は、送信内容がサービス改善に使われる場合があります。日記という個人的な内容を扱うため、この前提を理解した上でご利用ください。気になる場合は各自でGeminiの有料枠（Billing有効化）への切り替えをご検討ください。

> ⚠️ `AUTH_SECRET`・`INVITE_CODE`・`ACCOUNT_ENCRYPTION_KEY`・`KV_REST_API_TOKEN` は十分に長いランダム文字列にし、`.env*` を**絶対にコミットしない**でください（`.gitignore`済み）。

## 今後の拡張案

> **より詳細な製品ロードマップは [docs/ROADMAP.md](docs/ROADMAP.md) にあります**（3本柱: ふたりの話し合い分析（実装済み）/ 書けない人向け体験の強化 / 日記の保管庫。実装に着手できる詳細度で記載）。

- **Appleジャーナル用ショートカットのワンタップ配布**: 現状は新規ユーザーごとに11ステップの手作業でショートカットを作る必要がある（Apple側にJournalの公開APIが無いため）。あらかじめ作成したショートカットをiCloud共有リンクとして配布できれば、タップ1回で追加できるようになる。
- **パスワードリセット機能**: 現状は自己申告のみ。メール等を使ったセルフサービスのリセットフローがあると安心。
- **招待コードの個別管理**: 現状は全員共通の1つの `INVITE_CODE`。招待ごとに個別のコード・使用回数・有効期限を管理できるようにする。
- **課金・サブスクリプション**: 現状は無料公開のみ。利用者が増えてきた場合の有料プラン・決済連携（Stripe等）。
- **日記の本保存**: 現状は外部アプリ保存が中心。将来は暗号化した端末内DBやクラウドへ日記本体を保存し、複数端末での同期・一覧・検索・カレンダー表示（現状の「サーバーには日記を一切保存しない」というプライバシー方針からの転換になるため、慎重な検討が必要）。
- **チャンク分割時の再サンプリング精度向上**: 現状はモノラル16kHzに固定変換している（下記参照）。将来的に音質を選べるようにする余地がある。
- **日記からのプロフィール更新提案**: 生成された日記から「プロフィールに追加しますか？」と提案する半自動フロー（現状は手動で「情報を追加」する運用。日記本体を保存しない設計を保ったまま、ユーザーの明示的な承認を経て追加する形を想定）。
- **話者の口ぐせ学習**: 過去の日記から文体を学習し、よりその人らしい文章に。
- **タグの自動整理 / 感情の可視化**: `feelings`・`tags` を使った振り返りダッシュボード。
- **Web Push リマインダー**: 「今日の日記を書きましょう」通知。
- **多要素/パスキー認証**: WebAuthn 対応。
- **不正アクセスの可視化**: レート制限は分散対応済み。次はブロック回数の記録・通知など、攻撃の兆候に気づける仕組み。
- **page.tsx のさらなる分割**: 画面コンポーネントは分離済み。次は保存アクション・修正依頼フローをカスタムフックへ抽出する余地がある。
- **オフライン録音キュー**: 電波が無い場所で録音 → 復帰時に自動で文字起こし。
- **多言語対応**: 文字起こし言語の自動判定と英語日記など。
